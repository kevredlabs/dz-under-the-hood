The Problem: the public internet can't do #3

Public Internet
  • BGP optimise la reachability, pas la vitesse
  • 10-15 hops ISP par paire de validators
  • Jitter imprévisible (50-200ms)
  • Chaque validator déduplique seul (×1000)
→
Conséquences
  • Wallclock fairness impossible
  • Slots manqués, votes désynchronisés
  • Market makers structurellement désavantagés
  • Spreads larges → pire prix pour tous

What is DoubleZero ?

  • Réseau fibre privé et décentralisé — overlay sur de la fibre sous-utilisée
  • Contributeurs fournissent bande passante + hardware, rémunérés en 2Z
  • Validators et RPC nodes se connectent via Edge DZDs
  • Optimise le trafic validator ↔ validator : blocs, shreds, votes

Les briques : VXLAN, BGP, VRF

VXLAN — Overlay L2

Encapsule des trames Ethernet dans UDP:4789. VNI 24-bit = 16M réseaux virtuels. Jumbo frames MTU 9000 pour absorber le ~50 bytes d'overhead.

BGP — Routage inter-DZD

iBGP avec ASN privé 65342. Quand un validator se connecte à un Edge DZD, sa route est propagée automatiquement à tout le mesh. IS-IS en interne pour la convergence rapide.

VRF — Isolation du trafic

Tables de routage isolées sur un même switch : dz-prod (blocs, shreds, votes) vs dz-mgmt (SSH, monitoring). Un spike monitoring ne peut pas affecter le consensus.

Leader produit un bloc

Split en shreds pour distribution via Turbine

2
Shreds entrent dans le mesh DZ

Via GRE tunnel (mode IBRL) — pas d'ISP, pas d'internet public

3
Edge Filtering (FPGA)

Dédup shreds + validation format + rate limiting — en hardware, à line-rate

4
Routage mesh fibre

2-4 hops sur fibre dédiée, MTU 9000, routage optimisé

5
Livraison aux validators

Jitter minimal → shreds arrivent en ordre → reconstruction rapide

6
Votes consensus propagés retour

Tower BFT via le mesh — voting window utilisée efficacement
