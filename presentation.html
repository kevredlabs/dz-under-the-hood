Le Problème : l'internet public ne peut pas faire le #3

Internet Public
  • BGP optimise la disponibilité, pas la vitesse
  • 10-15 sauts FAI par paire de validateurs
  • Jitter imprévisible (50-200ms)
  • Chaque validateur déduplique seul (×1000)
→
Conséquences
  • Équité temporelle impossible
  • Slots ratés, votes désynchronisés
  • Market makers structurellement désavantagés
  • Spreads larges → prix moins bons pour tous

Qu'est-ce que DoubleZero ?

  • Réseau fibre privé décentralisé — overlay sur des fibres sous-utilisées
  • Les contributeurs fournissent bande passante + matériel, récompensés en 2Z
  • Validateurs et nœuds RPC se connectent via des Edge DZDs
  • Optimise le trafic validateur ↔ validateur : blocs, shreds, votes

Briques de Base : VXLAN, BGP, VRF

VXLAN — Overlay L2

Encapsule les trames Ethernet dans UDP:4789. VNI 24 bits = 16M réseaux virtuels. Jumbo frames MTU 9000 pour absorber le surcoût de ~50 octets.

BGP — Routage Inter-DZD

iBGP avec ASN privé 65342. Quand un validateur se connecte à un Edge DZD, sa route est automatiquement propagée sur tout le mesh. IS-IS en interne pour une convergence rapide.

VRF — Isolation du Trafic

Tables de routage isolées sur le même switch : dz-prod (blocs, shreds, votes) vs dz-mgmt (SSH, monitoring). Un pic de monitoring ne peut pas affecter le consensus.

Le Chemin d'un Bloc dans le Mesh

1
Le leader produit un bloc

Découpé en shreds pour la distribution via Turbine

2
Les shreds entrent dans le mesh DZ

Via tunnel GRE (mode IBRL) — aucun saut FAI, pas d'internet public

3
Filtrage Edge (FPGA)

Dédup shreds + validation format + rate limiting — en matériel, au débit ligne

4
Routage dans le mesh fibre

2-4 sauts sur fibre dédiée, MTU 9000, routage optimisé

5
Livraison aux validateurs

Jitter minimal → shreds arrivent dans l'ordre → reconstruction rapide du bloc

6
Les votes de consensus remontent

Tower BFT dans le mesh — fenêtre de vote utilisée efficacement
